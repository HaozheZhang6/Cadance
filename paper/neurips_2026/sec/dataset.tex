\subsection{Dataset}

\paragraph{Design principles.} BenchCAD rests on four principles.
\emph{(P1) Expert-generated geometry}: every family is hand-crafted by domain experts directly from industrial standards, who solve the standard-mandated geometric equations to produce parameterised CAD models that strictly respect engineering conventions and inter-parameter constraints.
\emph{(P2) Standard-table anchoring}: where a part has an industrial counterpart, parameters sample from real specification tables (ISO 22 V-belt cross-sections, DIN 338 twist-drill diameters, ISO 23509 bevel-gear pitch--module relations); 49\% of BenchCAD families (52/106) are standard-anchored, drawing from 47 unique ISO/DIN/EN/ASME/IEC codes (full code list in Appendix~\ref{app:standards}).
\emph{(P3) Family-level taxonomy}: records are grouped into 106 named families (\texttt{coil\_spring}, \texttt{helical\_gear}, \texttt{twisted\_drill}, $\ldots$), each implemented by a small Python module exposing a typed parameter schema, sampler, validator, and deterministic builder --- making coverage measurable and per-family analysis trivial.
\emph{(P4) Operation breadth}: BenchCAD exercises 49 distinct CadQuery operations spanning primitives, 2D sketching, advanced solid ops (helical sweeps, lofts, shells), boolean composition, holes, arrays, and finishing features --- covering CadQuery's core API surface, including operations (\texttt{helix}, \texttt{twistExtrude}, \texttt{polarArray}) that are rare or absent in DeepCAD/Fusion360-derived corpora.

\paragraph{CadGen pipeline.} For each family$\times$difficulty bucket we sample parameters under standard-table constraints, emit the CadQuery program, and sandbox-execute it (quarantining parse errors, $>30$\,s timeouts, and zero/inverted-volume outputs); every surviving render is then routed past a domain expert for visual sign-off, and only records passing all stages enter the release (17{,}900 as of May 2026).

\paragraph{Three released datasets.} \emph{BenchCAD} (17{,}900 verified CadQuery code parts; primary evaluation set), \emph{BenchCAD-QA} (2{,}400 paired image/code numeric-QA items), and \emph{BenchCAD-Edit} (748 curated edit pairs across dimensional, additive, subtractive, and multi-step categories). Each family supports easy/medium/hard tiers defined by parameter complexity. Every release ships a Croissant 1.0 metadata file~\citep{Croissant2024} validated by the public checker, hosted on Hugging Face under \texttt{BenchCAD/cad\_bench} with verification-pipeline source under MIT and data under CC-BY-4.0. Full datasheet (Appendix~\ref{app:datasheet}) and per-family schemas accompany the release.

\subsection{Tasks}
\label{sec:tasks}

BenchCAD evaluates models on four tasks, each targeting a distinct capability: \textsc{Vision QA} (geometric reasoning from rendered views), \textsc{Code QA} (symbolic understanding of CadQuery programs), \textsc{Edit Code} (instruction-following program editing), and \textsc{Vision2Code} (end-to-end synthesis of executable code from images).

\paragraph{\textsc{Vision QA} and \textsc{Code QA}.}
\label{sec:capability}
We organise the QA bank along a four-level capability hierarchy (Figure~\ref{fig:capability_hierarchy}), from low-level perception at the base to high-level synthesis at the apex:
\emph{(i) $L_1$ Holistic Visual Recognition} --- recognize the part family from multi-view renders and integrate views into a coherent volumetric understanding;
\emph{(ii) $L_2$ CAD Operations Understanding} --- read CadQuery (or infer from geometry) and map operations to features in the correct execution order;
\emph{(iii) $L_3$ Industrial Parametric Abstraction} --- abstract observed geometry into the parametric structure an engineer would write, respecting standard-table relations and engineering conventions;
\emph{(iv) $L_4$ Spatial/Code Reasoning} --- compose the lower three levels into syntactically valid, executable parametric code.
The matched-pair design isolates the levels: a large \textsc{Vision QA}--\textsc{Code QA} gap on the same questions reveals the cost of replacing direct code access with visual recognition (the \textbf{Holistic Spatial and Detailing Deficit}, \S\ref{sec:exp_failure_analysis}); low absolute \textsc{Code QA} flags a \textbf{CAD Operational Blindspot} at $L_2$.

\paragraph{\textsc{Code Edit}.} Given an original CadQuery program, a natural-language edit instruction, and a target STEP, the model must return a minimally-edited program whose rendered solid matches the target. Items span five task types T1--T5 (literal replace / chain transform / relative compute / feature edit / geometry rebuild; Table~\ref{tab:task_taxonomy}). We report \textbf{\texttt{mean\_norm}} (headroom-normalised improvement over the original-to-target gap, Eq.~\eqref{eq:edit_acc})as the headline since it controls for varying per-pair difficulty, while the others disambiguate ties and attribute failures to parametric exactness, geometric closeness, or syntactic execution. Edit-pair construction protocol and the F01--F08 failure taxonomy used in the per-model analysis are in Appendix~\ref{app:edit_protocol} (Table~\ref{tab:failure_codes}).



\paragraph{\textsc{Vision2Code}.} Generates executable CadQuery code from four canonical orthographic views. We report \textbf{IoU} (primary; rendered prediction vs.\ GT occupancy grid), \textbf{Chamfer distance}, \textbf{Feature score} (fillets, chamfers $\ldots$), essential-op recall (whether the model can apply key CAD operations), and \textbf{execution rate}. IoU is the headline; the other four disambiguate ties and attribute failures to geometric, structural, vocabulary-level, or syntactic causes. Question-bank construction, edit-pair protocol, rotation-invariant IoU (used for failure analysis only), and a scale-invariance ablation appear in Appendices~\ref{app:qbank}, \ref{app:edit_protocol}, \ref{app:iou}, \ref{app:ablation}.
